01목적 · 범위 · 사용 방법

1.1 목적

본 WBS는 SafeDoc Global Service의 5/15 데모일까지 잔여 작업과 그 이후 프로덕션 출시 작업을 단일 트리로 가시화하여, PM·엔지니어링·디자인·법무·CS·한국 본사 간 책임 경계를 명확히 한다. 작업 누락·중복·종속 누락을 방지하고, 분기별 슬립 리스크를 사전에 식별한다.

1.2 범위

  • 포함: 데모 마무리 작업, Next.js 프로덕션 이관, 디자인 시스템 코드화, 한국 API 연동, 법규 컴플라이언스, 데이터·분석, 운영·CS, Phase 2 후보(Stripe·네이티브·실시간 슬롯).
  • 제외: 한국 본사 측 API 변경, 한국 어드민 UX 재설계, 한국 병원 자체 운영 변경 (한국 본사 별도 트랙).

1.3 WBS 표기 규칙

  • ID 형식: W-M{milestone}-{branch}.{sub}.{leaf} — 예: W-M3-1.3.2 (M3 데모 분기 1, 서브 3, leaf 2).
  • 각 leaf는 1개 산출물(deliverable)을 가진다. 산출물 없는 leaf는 분기(branch)로 격상하거나 삭제.
  • 추정 단위: h(시간) 또는 d(8h 기준 일). 8h 미만은 h, 그 이상은 d.

1.4 RACI 약식

역할표기설명
R (Responsible)R실제 작업을 수행하는 사람. leaf당 1명을 원칙.
A (Accountable)A최종 결과에 대한 책임자. 결정 권한 보유. leaf당 정확히 1명.
C (Consulted)C수행 전·중 의견 청취 대상.
I (Informed)I결과만 통보받는 대상.

1.5 상태 정의

표기의미다음 단계
DONE완료, 산출물 검증됨—
IN-PROGRESS진행 중, R 명시완료 시 DONE
PENDING계획됨, 미착수R 배정 → IN-PROGRESS
BLOCKED외부 종속으로 보류해소 즉시 IN-PROGRESS
DEFERREDPhase 2로 이관M7+에서 재평가

A부록 A — 작업 ID 명명 규칙

모든 WBS leaf는 다음 패턴을 따른다:

W-M{milestone}-{branch}.{sub}.{leaf}
   │       │        │     │
   │       │        │     └─ 최하단 작업 번호 (1, 2, ...)
   │       │        └─ 분기 내 서브 그룹 (1, 2, ...)
   │       └─ 최상위 분기 (1=데모, 2=프로덕션, 3=데이터베이스 & 운영자 계약, 4=법규, 5=데이터, 6=운영, 7=Phase 2)
   └─ 마일스톤 (M3=데모, M4=프로덕션 킥오프, M5=Next.js + DB 마이그레이션, M6=운영자 계약 + 베타, M7=Stripe)

예: W-M3-1.3.2 = M3 데모 분기의 디자인 QA 서브의 두 번째 leaf(모바일 회귀).

B부록 B — 상태 라이프사이클

각 leaf의 상태는 단방향으로 진행된다(DEFERRED ↔ PENDING은 예외 양방향).

[PENDING]  ──(R 배정)──→  [IN-PROGRESS]  ──(산출물 검증)──→  [DONE]
   │                            │
   ↓ (외부 종속)                  ↓ (외부 종속)
[BLOCKED]  ──(해소)──→  [IN-PROGRESS]
   │
   ↓ (스코프 변경)
[DEFERRED]  ──(M7+에서 재평가)──→  [PENDING]

상태 전이는 R 또는 A가 기록한다. BLOCKED → IN-PROGRESS 전이는 차단 사유 해소 evidence(예: 변호사 회신 메일 ID)를 같이 남긴다.

본 WBS는 5/15 데모 이후 매주 PM 주관으로 갱신된다(변경은 v1.2 → v1.x 패치). 분기별 leaf 추가/삭제는 본 문서에 기록 후 RACI 통보. 임계 경로 변경 시 §04 즉시 업데이트. 본 문서가 다른 .pm/ 문서와 충돌할 경우, 본 문서를 우선한다.
