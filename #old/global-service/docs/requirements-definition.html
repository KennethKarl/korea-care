01문서 개요

1.1 목적

본 문서는 SafeDoc Global Service의 v1 데모(2026-05-15) 범위에 포함되는 모든 기능적·비기능적 요구사항을 단일 출처로 정의한다. 본 문서는 합의된 .pm/product-brief.md(2026-04-21 확정), .pm/design-brief.md, .pm/korean-data-mapping.md, 그리고 인터랙티브 프로토타입(prototype/demo.html)에서 검증된 사양을 통합한다.

1.2 적용 범위

  • 본 문서는 2026-05-15 CEO 데모 출시 범위에 한정된다.
  • 결제(Stripe), 네이티브 앱, 실시간 슬롯 예약, 환불 처리 등은 Phase 2로 명시 분리된다.
  • 의료 기록(EHR/EMR)·진료 기록·임상 의사결정 지원은 영구적으로 본 제품의 범위가 아니다.

02제품 정의

2.1 한 줄 정의

SafeDoc Global Service는 미국 거주자가 한국의 SafeDoc 제휴 의료기관을 진료과 단위로 탐색하고, 본인의 시민권·기업 제휴에 따라 차등화된 가격을 확인하여, 예약 요청과 부가서비스(숙박·항공·투어)를 통합 제출할 수 있는 이중언어(EN/KO) 웹 플랫폼이다.

2.2 본 제품이 아닌 것 (Anti-scope)

  • EHR/EMR 시스템 아님
  • 원격진료(Telehealth) 아님
  • 임상 기록 시스템 아님
  • 실시간 슬롯 예약 시스템 아님
  • 의료 진단/처방 도구 아님
  • 환자-의사 메시징 채널 아님

2.3 가치 제안

미국 거주 사용자

큐레이션된 한국 상급종합병원 네트워크에 영문 인터페이스로 접근. 투명한 USD 가격, 단일 코디네이터(SafeDoc)가 예약과 여정을 일괄 응대.

제휴 기업

임직원 복지로 보조금 적용된 선택 진료를 제공할 수 있는 채널. 기업 코드 발급 → 어드민 검증 → 부서별 할인율 적용.

SafeDoc

기존 한국 병원 계약·DB·어드민을 그대로 재사용해 미국 시장 성장 채널을 확보.

03사용자 정의

3.1 주요 페르소나 (3종)

FOREIGNER

미국 일반 거주자

미국 시민권자, 비-한국 시민권. 외국인 단가 노출. 단일 코디네이터와 영어 응대 필요. 예: 텍사스 거주 미국인.

KR_RESIDENT

한인 (한국 시민권 보유)

미국 거주 한국 시민권자(이중국적 포함). 한국 내국인 단가 노출. 자가 신고 기반.

KR_RESIDENT + Enterprise

기업 제휴 사용자

한국 시민권 + 가입 시 ACME 등 기업 코드 입력 → 어드민 검증 후 부서별 추가 할인 스택.

3.2 시민권 자가 신고 정책

  • v1 범위에서는 여권/시민권 서류 검증을 수행하지 않는다.
  • 가입 시 KR / non-KR 두 옵션 중 자가 신고. 가격 티어가 즉시 잠긴다.
  • 허위 신고 시 결제·치료 단계의 검증은 한국 측 어드민·CS 흐름에 위임 (Phase 2 이후 정책 강화).

3.3 계정 유형

유형가입 입력가격 적용오프보딩
Individual OAuth(Google/Apple) + 시민권 자가 신고 표준(KR_RESIDENT 또는 FOREIGNER) 사용자 본인 탈퇴
Enterprise-affiliated 위 + 기업 코드 입력 → "검증 대기" 상태 검증 완료 시 (기업, 진료과)별 할인율 자동 스택 기업이 SafeDoc에 통보 → 어드민에서 권한 회수

4.3 공통/위젯 요구사항

FR-COMMON-1 — 페르소나 패널 (데모 전용)

플로팅 위젯에서 3개 페르소나(us_individual / ka_kr_citizen / ka_enterprise) 간 즉시 전환. 전환 시 가격/온보딩/요금 표시가 라이브로 변경된다. 프로덕션에는 반영되지 않는 데모 전용 컨트롤. 자세히는 §4.5 참조.

FR-COMMON-2 — 언어 토글 (EN / KO)

  • 헤더 우측 세그먼트 컨트롤. 세션 동안 선택 유지(localStorage).
  • 모든 UI 문자열은 T 번역 맵에 키로 정의({en, ko}). 누락된 키는 영문 폴백.
  • :lang(ko)에서 라인하이트 1.65 적용.

FR-BROWSE-1 — 둘러보기 서브내브 (Browse tabs)

  • "Specialties (Hospitals)" 탭(#/departments, #/hospitals) ↔ "Treatments" 탭(#/services) 간 전환.
  • 스코핑 규칙: Treatments 탭에서 진료과를 선택하면 병원 필터가 해당 진료과 소속 병원으로 자동 한정된다.
  • 필터 레일은 두 탭이 공유: 진료과 · 병원 · 가격대(Any / <$500 / $500–1.5K / $1.5K–3K / $3K+) · 고민(concern) · 정렬(추천/예약많은순/가격↑↓/짧은방문순).

FR-CART-1 — 멀티-병원 멀티-시술 장바구니

  • 같은 시술 중복 추가 불가(토스트 "이미 장바구니에 담겨 있습니다").
  • 라인 아이템마다 시술 단가 · 방문 날짜 · 병원 표시. 가격은 사용자 티어가 반영된 실시간 값.
  • 같은 (병원, 방문 날짜) 페어는 1개 예약번호로 자동 묶음. 예약번호 분리 안내를 사용자에게 명시.
  • 합계: 시술 합계 · 총 할인 금액 · 예상 총 결제 금액 라이브 계산.
  • 모든 라인의 방문 날짜가 입력되어야 결제(#/checkout) 진행 가능.

FR-PRICING-1 — 할인 충돌 해소 (Conflict Resolution)

  • 비-스택 쿠폰(NEW200 등)과 기업 할인이 동시에 적용 가능한 상태가 되면 사용자에게 라디오로 양자택일을 강제한다.
  • 선택 결과는 state.conflictChoice에 보존되어 새로고침 후에도 유지.
  • 기본 선택은 강한 할인이 자동으로 미리 선택되나, 사용자가 명시적으로 다른 쪽으로 변경 가능.

FR-SIM-1 — 인터랙티브 가격 시뮬레이터 (랜딩 하단)

  • 위치: #/ 랜딩 하단 풀 섹션(id="simulator"). 모바일/데스크탑 모두 렌더링.
  • 5개 레버 — (1) 서비스 선택(병원별 옵트그룹), (2) 로그인 상태 토글, (3) 시민권(KR/non-KR), (4) 기업 상태(none/pending/verified), (5) 쿠폰(빈값/SAFEDOC10/NEW200).
  • 4단계 슬라이더(Guest → Signed in → +Enterprise → +Coupon)로 시나리오 캐스케이드 이동 가능.
  • 우측 영수증: 기본 단가(strike-through) · 티어 적용 · 기업 할인 · 쿠폰 할인 · 예상 총액 · 절약 금액(%) · 진행률 바.
  • 비회원 상태에서는 가격이 잠금(스트라이크) + "Sign in to view" 칩 노출.
  • 충돌 조건 발생 시 라디오로 즉시 선택(앱 본체와 동일한 동작).
  • 비-부작용: 시뮬레이터의 상태(simState)는 앱 본체의 state와 분리되며, 네비게이션 이탈 시 초기화. 실제 카트·예약·로그인 상태를 변경하지 않는다.

4.4 핵심 사용자 플로우

플로우 1 — 가입 (첫 방문)

  1. EN/KO 홈 진입. 언어는 자동 감지 + 수동 토글 가능.
  2. 계정 없이 병원·시술 탐색 가능 (게스트 가격은 표시되나 "Sign in to view" 잠금 상태 - 시뮬레이터 참조).
  3. "Book" 또는 "See my rate" 또는 "Add to cart → Checkout" 클릭 시 Google/Apple 로그인 유도(?next= 콜백 보존).
  4. OAuth 콜백 후 1회성 프로필 입력: 시민권(KR/non-KR), 선택적 기업 코드(예: ACME-2026), 연락 이메일(자동 채움).
  5. 기업 코드는 "검증 대기"(pending) 상태. 검증 전까지는 자가 신고 시민권 티어 가격이 적용됨. 데모에서는 "관리자 승인 시뮬레이션" 버튼으로 verified로 즉시 전이 가능.

플로우 2 — 탐색 & 발견 (Specialties / Treatments 듀얼 진입)

  1. 홈 → 진료과 선택(피부과·정형외과·안과·성형외과·치과·건강검진·내과·산부인과 8종) 또는 시술 카탈로그(#/services)에서 시술 단위로 직접 탐색.
  2. 진료과 진입 시: 진료과로 필터링된 병원 목록 (카드 = 병원명 EN/KO, 위치, 평점, 사용자 티어 가격 범위).
  3. 시술 진입 시: 진료과 + 병원 + 가격대 + 고민(concern) + 정렬 필터로 시술 카드 그리드 탐색.
  4. 병원 상세 → 서비스/시술, 시술별 가격(티어 적용), "예약하기" 또는 "장바구니 담기".

플로우 3 — 장바구니 → 결제 (멀티-예약)

  1. 시술 카드 또는 시술 상세에서 "Add to cart". 동일 시술 중복 추가 불가.
  2. 카트(#/cart)에서 라인별 방문 날짜 선택. 같은 (병원, 날짜) 페어는 1개 예약번호로 자동 묶음.
  3. 모든 라인의 방문 날짜가 입력되어야 "Proceed to checkout" 활성화.
  4. Checkout(#/checkout)에서 1인 환자 프로필 입력 → 모든 예약에 공통 적용.
  5. "Confirm & submit all reservations" → 예약 그룹별 WAITING 상태 생성 → #/checkout-done 진입.

플로우 4 — 단건 예약 요청 (직접 진입)

  1. 병원 상세에서 "Book" 클릭 시 단건 예약 폼(#/book/:hospital/:service)으로 직접 진입.
  2. 필수: 이름, 이메일, 전화, 나이, 성별, 국적, 여권번호, 희망 진료과(자동 채움 · readonly), 희망 방문일, 한국 입국일, 한국 출국일.
  3. 선택: 호텔/지역, 항공편 번호, 추가 메모.
  4. 부가서비스(체크박스): 숙박, 항공, 투어. 자유 메모 1개 공통 필드.
  5. 가격 요약(라이브): 티어 단가 → 기업 할인 → 쿠폰 → 충돌 시 라디오로 강제 선택.
  6. 제출 → "병원 확인 대기"(WAITING) 상태로 저장. 5/15 데모는 여기까지 (결제 X).

플로우 5 — 병원 확인 (Out-of-band)

  1. 병원이 한국 어드민에서 대기 요청을 확인.
  2. 승인 시 "확인됨"(CONFIRMED) 상태로 전이 → 사용자 화면에 즉시 반영.
  3. 거절 또는 일자 변경 제안 시 사용자에게 알림.
  4. 데모에서는 예약 상세(#/booking/:id) 화면에 노출되는 "병원 확정 시뮬레이션" 버튼으로 즉시 CONFIRMED 전이 가능 (§4.5 참조).

플로우 6 — 부가서비스 CS 상담 (Out-of-band)

  1. 부가서비스 포함 예약은 SafeDoc 어드민의 CS 큐에 노출.
  2. CS팀이 Slack 채널로 사용자와 이메일/플랫폼 메시지로 응대.
  3. 상담 노트는 어드민에서 해당 예약에 첨부 로깅.

6.3.1 클라이언트-사이드 검증 규칙

  • 여권번호: 정규식 ^[A-Z0-9]{6,12}$ 위반 시 오류 "여권 번호는 영문·숫자 6~12자리여야 합니다."
  • 날짜 일관성: exitDate > entryDate 위반 시 오류 "출국일은 입국일 이후여야 합니다."
  • 이메일: HTML5 type="email" 기본 검증.
  • 나이: 18 이상 120 이하 정수.
  • 서버 검증은 BFF에서 동일 규칙 + 입력값 길이/문자 sanitize(필드 allowlist).

6.4 수집하지 않는 데이터 (명시적 제외)

본 제품은 다음을 수집·저장·표시하지 않는다 (HIPAA 노출 최소화 의도):
  • 임상 노트(clinical notes), 진단명(diagnoses), 증상 기록(symptoms)
  • EHR/EMR 데이터, FHIR 리소스, CCDA 문서
  • 실험실 검사값(lab values), 영상 검사(imaging), 병리(pathology)
  • 처방(prescriptions), 투약 이력(medication history)
  • 보험 정보(insurance), 청구 코드(billing codes), 의사-환자 메시지
  • 업로드된 의료 문서(PDF/이미지/CCDA), 의료 이미지(DICOM)

예약 요청은 "특정 진료과 상담 의향"이며, 임상적 만남이 아니다. SafeDoc은 Covered Entity 또는 Business Associate 관계를 형성하지 않는다.

6.5 규제 함의

  • 여권번호는 강한 PII이며 일부 주법상 별도 보호 대상.
  • 진료과 의향 + 연락처는 연방 HIPAA의 PHI는 아니지만, TX HB 300, CCPA SPI 정의상 건강정보로 분류될 수 있음.
  • 상세는 .pm/legal/hipaa-position.md Option C(non-HIPAA + TX HB 300 + CCPA/CPRA + PIPA) 참고.
